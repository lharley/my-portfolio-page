About Me

I believe good UX design is the result of hard work and insight born of knowledge. UX design is about solving problems, not making things look good. The more you know and understand a problem, the easier it is to define a solution. Following a systematic process is important, but all artifacts in the UX lifecycle are exercises in gaining knowledge about users and understanding the task context. I love designs that are clean and simple, not because that's the current fashion, but because clean and simple focuses the user on the essentials and doesn't waste the user's time. Simple feels like it should be easy to do, since, well, it's simple. But simple turns out to be very hard to do. Simplicity only comes from knowing the user and the task space. If you have knowledge, you know what is clutter and what is essential to the user, so you can design an experience that allows the user to accomplish the task goals, with speed, few errors, and satisfaction.

So, if I'm so passionate about design and users, why am I a front-end developer? One reason is that I aspire to be the UX/UI front-end developer I always wanted to work with. I am sensitive to the details of graphic and interaction design and I know UI platform standards and UX best practices in a way that most front-end developers do not. This means my implementations reflect the spirit of an UX design, not just the functional specifications. They provide better UX experiences, because they follow UX best practices in terms of visual design, information design, and interaction design. A second reason that I code is that coding provides knowledge and insight into Web technologies, as well as the technical challenges developers face. This knowlege makes me a better designer and UX problem solver. Coding also makes me a better team collaborator, with developers and product stakeholders. Besides, making Web pagse come alive with interactions, insightful data, and elegant animations is just plain fun.

MORPHOTRUST USA

Issuance Manager 360 - Supervisor Dashboard

As the information architect and lead UX desgner, I lead the re-envisioning of the company's main product, renamed Issuance Manager 360.

For the re-envisioning of the company's main product, thirty-two individual applications and device drivers were combined into a modern, modularized platform. Individual modules, such as Identity 360, Issuance 360, and Security 360, had to function as a coordinated application or work as an individual, stand-alone applications, depending on which modules a customer purchased.

As part of the re-envisioningin we modernized the software to include both a Web application and Web-enabled mobile apps. This expansion allowed us to address user issues in new ways. In user research, we found that the email alert for critical issues was not alerting in a timely manner. Users only checked email a few times a day. The staff constantly used the issuance software through the workday, so I proposed a dashboard for the new mobile-enabled Web app, where each user's critical job functions were provided a status and an alert warning, if needed. In addition to the email notifications, text message (SMS) alerts were sent directly to user's mobile devices.

MIT LINCOLN LABS

Resources Manager

Mockup of a specialized app that served as a companion to the division's main Website.

MIT Lincoln Labs is a secure facility. The mockups shown have been adapted from the actual design I prepared while working at the Lab. The functional concepts and the UI design are the same, but the content has been generalized and cleaned of any specific reference to Lab personnel, project names, project topics, equipment, and lab capabilities and processes.

The division I worked for managed several facilities for use in researching, developing, and testing engineering prototypes. I was responsible for the design and development of the division's website, where in addition to other business, the resources could be booked, equipment reserved, and project participants invited. Lab personnel worked in many locations throughout the Lab and Hanscom AFB. They wanted a way to access and manage resources more conveniently, namely on mobile devices. Because the division's website was was large and served many purposes, I proposed to extract the functionality for resource scheduling and create an app solely dedicated to resources scheduling and management.

The mockups were prepared using Adobe XD and Illustrator. There is a PDF of the proposed design. There is also a Click Thru Prototype of the mockups on Adobe Cloud, which you can access via smartphone or tablet.

Pfizer Inc.

Presentation at Software Improvement Offsite

Pfizer leads wanted an intro as a prelude to discussion on incorporating UX into dev process. Reflects my views on UX design process.

I am often asked to explain UX and User Centered Design. Part of the role of an UX Designer -- or anyone who works in a sub-disciplines -- is to advocate for and educate others about the UX discipline. As Kim Flarerty of the Nielson Norman Group states, "good UX design, as is a very abstract, analytical discipline." As such it can be hard for others to understand. Even people who call themselves UX designers don't always understand the deeply analytical nature of the work. Because there are elements of visual design in the work and because UX designers talk of "empathy" and "user delight," the discipline can seem superficial and vague. It is not.
